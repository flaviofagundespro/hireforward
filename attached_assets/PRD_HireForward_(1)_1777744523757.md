## 1. Visão e Problema

### O problema central
Processos seletivos ainda avaliam "o profissional de 2015" — testam memória de sintaxe, não capacidade de resolução de problemas. Plataformas como CoderPad e HackerRank bloqueiam ferramentas e medem o que é fácil de medir, não o que importa.

### A premissa nova (não uma feature — uma inversão de paradigma)
> **Contrate quem resolve problemas, não quem decora respostas.**

O candidato tem acesso livre a qualquer ferramenta. Um agente de IA conduz a entrevista, avalia o processo de raciocínio, o uso de ferramentas e a qualidade da solução — com base em rubricas definidas pela empresa. Funciona para qualquer área: Engenharia, Vendas, Marketing, Financeiro, CS, etc.

---

## 2. Usuários e Papéis

| Papel | Descrição |
|---|---|
| **Owner (você)** | Admin da plataforma. Vê todas as empresas, uso de tokens e custos. |
| **Admin RH** | Configura processos seletivos, agentes entrevistadores, critérios de avaliação e visualiza resultados. |
| **Gestor** | Acessa relatórios dos processos vinculados ao seu time. Não configura nada. |
| **Candidato** | Recebe um link único, interage com o agente entrevistador. Não tem conta na plataforma. |

## 4. Arquitetura de Multi-tenancy

Cada empresa (tenant) é isolada. Toda tabela tem `company_id`. O candidato acessa via token único no link — sem criar conta.

```
companies
  ├── users (RH admins e gestores)
  ├── job_processes (processos seletivos)
  │     ├── evaluation_criteria (rubricas)
  │     ├── interviewer_config (personalidade, área, perguntas-guia do agente)
  │     └── candidates
  │           ├── interview_sessions (conversa completa com o agente)
  │           └── evaluations (score por critério + justificativa)
  └── token_usage (log de uso por processo)
```

## 5. Módulos do MVP

---

### 5.1 Módulo: Configuração do Agente Entrevistador

**Objetivo:** Deixar o RH configurar o "entrevistador ideal" sem precisar saber nada de IA.

**Fluxo:**
1. RH cria um novo processo seletivo (nome da vaga, área, nível de senioridade).
2. Um **agente configurador** (AI) entra em conversa com o RH via chat:
   - "Quais são as 3 competências mais importantes para esse cargo?"
   - "O candidato pode usar IA e internet durante a entrevista?" (default: sim)
   - "Prefere uma entrevista técnica, comportamental ou híbrida?"
   - "Quer que o agente simule um cenário real da empresa?"
3. A partir das respostas, o sistema gera automaticamente:
   - **System prompt do agente entrevistador**
   - **Rubricas de avaliação** (critérios com peso e descritores de 1–5)
   - **Roteiro de perguntas** (adaptável em tempo real pelo agente)
4. RH revisa e aprova — ou pede ajuste ao configurador.
5. Sistema gera **link único por candidato**.

**Entidades geradas:**
```json
{
  "job_process_id": "uuid",
  "title": "Analista de Growth",
  "area": "Marketing",
  "seniority": "Pleno",
  "tools_allowed": true,
  "interview_type": "hybrid",
  "agent_system_prompt": "Você é um entrevistador sênior de Growth...",
  "criteria": [
    { "name": "Raciocínio analítico", "weight": 30, "descriptors": {...} },
    { "name": "Uso inteligente de ferramentas", "weight": 25, ... },
    { "name": "Comunicação", "weight": 20, ... },
    { "name": "Conhecimento de domínio", "weight": 25, ... }
  ]
}
```

---

### 5.2 Módulo: Entrevista do Candidato

**Objetivo:** Interface limpa, acessível por link único, sem criação de conta.

**Fluxo:**
1. Candidato acessa `hireforward.com/i/[token_unico]`
2. Tela de boas-vindas: nome da empresa (logo), cargo, instruções ("você pode usar qualquer ferramenta durante esta entrevista")
3. Interface de chat com o agente — campo de texto + enviar. Simples e focado.
4. Agente conduz a entrevista de forma conversacional:
   - Apresenta-se, explica o processo
   - Faz perguntas adaptativas (não segue script rígido)
   - Pode pedir que o candidato compartilhe código, raciocínio, links ou qualquer artefato via texto
5. Duração estimada exibida (ex: "~30 minutos")
6. Ao final, agente encerra e agradece. Candidato não vê score.
7. Toda a conversa é salva e enviada para avaliação.

**Regras:**
- Link expira após uso ou após prazo definido pelo RH
- Sessão tem timeout configurável (ex: 60 min)
- Sem restrição de ferramentas — candidato usa o que quiser

---

### 5.3 Módulo: Avaliação Automática

**Objetivo:** Após o fim da entrevista, um segundo agente avalia a conversa contra as rubricas.

**Fluxo (background, após encerramento):**
1. Sistema envia para o agente avaliador: `[system_prompt_avaliador] + [transcrição completa] + [critérios e pesos]`
2. Agente retorna JSON estruturado:
```json
{
  "candidate_id": "uuid",
  "overall_score": 78,
  "recommendation": "Avançar para próxima etapa",
  "criteria_scores": [
    {
      "name": "Raciocínio analítico",
      "score": 4,
      "max": 5,
      "justification": "Candidato decompôs bem o problema de CAC, usou estrutura MECE..."
    },
    ...
  ],
  "highlights": ["Uso criativo de IA para análise de dados", "Perguntou sobre contexto antes de responder"],
  "red_flags": ["Não validou hipótese central antes de avançar"],
  "summary": "Candidato demonstrou raciocínio sólido e uso maduro de ferramentas..."
}
```
3. Score e relatório ficam disponíveis no dashboard do RH imediatamente.

**System prompt base do agente avaliador:**
```
Você é um avaliador sênior de processos seletivos. Receba a transcrição 
de uma entrevista e os critérios de avaliação com seus pesos. 
Avalie cada critério de 1 a 5 com justificativa específica baseada 
em evidências da conversa. Não infira o que não foi dito. 
Retorne apenas JSON válido no formato especificado.
```

### 5.4 Dashboard — RH da Empresa

**Seções:**

#### Visão Geral (Home)
- Cards: Total de candidatos avaliados | Processos ativos | Processos encerrados | Taxa de aprovação média
- Gráfico de linha: candidatos avaliados por semana (últimas 8 semanas)
- Tabela: processos ativos com barra de progresso (X de Y candidatos avaliados)

#### Processos Seletivos
- Lista de todos os processos (cards ou tabela)
- Status: Rascunho / Ativo / Pausado / Encerrado
- Ações: Ver candidatos | Pausar | Gerar link | Duplicar processo

#### Candidatos (dentro de um processo)
- Tabela com: Nome | Score | Recomendação | Data | Ações
- Filtros: score mínimo, recomendação, data
- Click abre o relatório completo do candidato

#### Relatório do Candidato
- Score geral com gauge visual
- Breakdown por critério (barras horizontais)
- Destaques positivos e red flags
- Transcrição completa da entrevista (expansível)
- Botão: "Gerar PDF do relatório" / "Exportar para CSV"

#### Relatório para Gestor (view simplificada)
- Gestor acessa via link ou login limitado
- Vê apenas os processos do seu time
- Dashboard simplificado: candidatos, scores, recomendações
- Não vê configurações do agente nem dados de outros times

### 5.5 Dashboard — Owner (você)

**URL:** `/admin` — protegido por credenciais separadas

**Seções:**

#### Empresas Cadastradas
- Total de empresas ativas / inativas / trial
- Tabela: Nome | Plano | Data de cadastro | Processos criados | Candidatos avaliados | Status
- Busca e filtros

#### Uso de Tokens (mensal e anual)
- Cards: Tokens usados no mês atual | Tokens usados no ano | Custo estimado mês (USD) | Custo estimado ano (USD)
- Gráfico de barras: uso por empresa (top 10)
- Gráfico de linha: evolução mensal do uso total
- Tabela de log: empresa | processo | candidato | tokens input | tokens output | custo | data

#### Cálculo de custo (claude-sonnet-4):
```
Input:  $3.00 / 1M tokens
Output: $15.00 / 1M tokens
Custo = (tokens_input * 0.000003) + (tokens_output * 0.000015)
```

#### Financeiro (futuro — placeholder no MVP)
- MRR estimado
- Churn
- LTV médio

## 6. Modelo de Dados (simplificado)

```sql
-- Empresas (tenants)
companies (id, name, logo_url, plan, status, created_at)

-- Usuários da empresa
users (id, company_id, email, role[admin|manager], created_at)

-- Processos seletivos
job_processes (id, company_id, title, area, seniority, status, agent_config jsonb, created_at)

-- Critérios de avaliação
evaluation_criteria (id, job_process_id, name, weight, descriptors jsonb)

-- Candidatos
candidates (id, job_process_id, name, email, invite_token, token_expires_at, status)

-- Sessões de entrevista
interview_sessions (id, candidate_id, started_at, ended_at, transcript jsonb)

-- Avaliações
evaluations (id, candidate_id, session_id, overall_score, recommendation, criteria_scores jsonb, highlights jsonb, red_flags jsonb, summary text, created_at)

-- Log de tokens
token_usage (id, company_id, job_process_id, candidate_id, tokens_input, tokens_output, cost_usd, created_at)
```

## 7. Fluxo Completo (end-to-end)

```
RH cria processo
      ↓
Agente configurador (chat) define critérios e system prompt
      ↓
RH aprova → sistema gera links únicos para candidatos
      ↓
Candidato acessa link → chat com agente entrevistador
      ↓
Entrevista encerra → agente avaliador processa transcrição
      ↓
Score + relatório aparecem no dashboard do RH
      ↓
RH filtra, compara e gera relatório para gestor
```

## 8. Requisitos Não-Funcionais

| Requisito | Especificação |
|---|---|
| Latência do agente | < 3s para primeira resposta (streaming preferível) |
| Sessão de entrevista | Suporta até 60 minutos de conversa |
| Isolamento de dados | company_id em todas as queries — nenhum tenant vê dados de outro |
| Link do candidato | Token JWT com expiração configurável |
| Exportação | Relatório em PDF e CSV |
| Mobile | Interface do candidato responsiva (muitos acessam pelo celular) |

## 10. Critérios de Sucesso do MVP

- RH consegue configurar um processo seletivo completo em menos de 10 minutos sem documentação
- Candidato consegue iniciar a entrevista sem criar conta, em menos de 60 segundos
- Avaliação automática gerada em até 30 segundos após fim da entrevista
- Owner consegue ver custo de tokens em tempo real no dashboard admin
- Pelo menos 1 empresa externa usa a plataforma e avalia um candidato real

### Notas de implementação para o Replit Agent
1. Use **streaming** (SSE) para as respostas dos agentes — entrevistas longas não podem aguardar resposta completa
2. O agente avaliador deve ser chamado em **background** após `interview_sessions.ended_at` ser setado
3. Toda chamada à API da Anthropic deve logar `usage.input_tokens` e `usage.output_tokens` na tabela `token_usage`
4. O link do candidato (`/i/[token]`) não requer autenticação — valide apenas o JWT do token de convite
5. Use `company_id` como middleware de validação em todas as rotas autenticadas
6. A interface do candidato deve ser **mobile-first** com campo de texto expansível
7. O dashboard do owner em `/admin` usa credenciais separadas (não é um tenant da plataforma)
